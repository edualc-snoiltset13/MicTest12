The Case for One-File Websites

Sometimes the right amount of tooling is none at all.

There is a quiet pleasure in opening a single .html file in a browser and seeing it work. No build step, no node_modules, no version mismatches between three frameworks. Just a document, styled and scripted in place, that any browser made in the last decade can render.

This page is an experiment in that idea — taking the patterns we normally split across components, partials, and modules, and collapsing them into a single artifact you can email to someone.

The tradeoffs

One-file pages have real limits. They don't scale to applications with many routes, they can't share components across pages cleanly, and they're awkward to maintain past a certain size.

"Use the right tool for the job — but recognize that 'no tool' is sometimes the right tool."

But for a portfolio, a landing page, a documentation index, or a quick prototype, the simplicity is genuine. The browser is doing the work you'd otherwise pay a framework to do.

A small example

function progress() {
    const h = document.documentElement;
    const pct = (h.scrollTop / (h.scrollHeight - h.clientHeight)) * 100;
    document.getElementById("progress").style.width = pct + "%";
}
window.addEventListener("scroll", progress, { passive: true });

That's the entire scroll-progress implementation at the top of this page. Eight lines. No library.

When to reach for more

The honest answer: as soon as you need persistent state across multiple URLs, server-side data, authentication, or a team workflow. Single-file pages are a starting point, not a destination — but they remain a useful one.

History

A Selective Timeline of the Web

Milestones that shaped how single-page documents look and behave today.

1991

The first website

Tim Berners-Lee publishes the first webpage at CERN — a plain HTML document describing the World Wide Web project.

1996

CSS arrives

CSS Level 1 becomes a W3C recommendation, separating presentation from structure for the first time.

2005

The Ajax era

XMLHttpRequest becomes mainstream after Gmail and Maps, kicking off the move toward dynamic single-page applications.

2014

HTML5 finalized

Native semantic elements, video, audio, canvas, and the <details> element used by this page's FAQ section.

2017

CSS Grid ships

Two-dimensional native layout becomes broadly available, replacing years of float-based hacks.

2024

Modern primitives

Container queries, :has(), view transitions, and scroll-driven animations bring previously-JS features into pure CSS.

Questions

Frequently Asked

Native <details> elements — click any question to expand it.

What is this page exactly?

A single HTML file that demonstrates a wide range of common web patterns — sticky navigation, articles, timelines, data tables, FAQs, and more — without using any external libraries or build tooling.

Can I use this as a template?

Yes. Copy the file, delete the sections you don't need, replace the content with your own, and adjust the CSS variables at the top to change the color scheme.

Why no JavaScript framework?

For static, single-page content the overhead of a framework outweighs the benefits. Browsers natively handle most of what's needed here — smooth scrolling, expandable details, responsive layout — without any libraries.

Will this work offline?

Yes. There are no external assets, fonts, or scripts. Open the file in any modern browser and everything renders locally.

How do I add a new section?

Copy any existing <section> block, give it a unique id, add a matching link to the nav at the top, and replace the content.
